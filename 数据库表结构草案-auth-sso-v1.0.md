## 1. 设计目标

本设计同时覆盖两层：

- 逻辑层：谁拥有什么数据，数据放在哪里
- 物理层：表、字段、主键、外键、索引、约束按什么规范落地

本版重点解决两个问题：

- `Portal / IdP / PostgreSQL / Redis` 的数据边界
- 数据库设计遵循常用 PostgreSQL 建模规范

---

## 2. 系统与存储关系

## 2.1 总体关系

```text
Browser
  -> apps/portal
       -> PostgreSQL.portal_core
       -> Redis.portal:*
       -> apps/idp
            -> PostgreSQL.idp_auth
            -> Redis.idp:*

Sub Application
  -> apps/idp
       -> PostgreSQL.portal_core.clients*
       -> PostgreSQL.idp_auth
       -> Redis.idp:*
```

## 2.2 数据归属

`Portal` 负责：

- 用户
- 部门
- 角色
- 权限
- 菜单
- Client 配置
- Portal Session
- 管理审计与登录日志

`IdP` 负责：

- 登录页 / 认证确认页 / 错误页
- OIDC 端点
- 授权码与 Token 运行态
- IdP Session
- Better Auth 内部认证持久化数据

## 4. 物理建模规范

这是本版相对之前文档的关键修正，以下规范为冻结规则。

## 4.1 主键规范

核心实体表统一使用：

- `id bigint generated by default as identity primary key`
- `public_id varchar(64) not null unique`

说明：

- `id` 是数据库内部主键，仅用于关系连接
- `public_id` 是接口层、日志、外部系统看到的字符串 ID
- API / PRD 中出现的 `u_1`、`r_1`、`d_1` 等，都对应 `public_id`

## 4.2 外键规范

- 所有关联统一使用内部 `id bigint`
- 不使用 `public_id` 做物理外键
- 不使用字符串编码字段做外键

## 4.3 关联表规范

多对多关联表统一使用：

- 组合主键
- 不额外创建代理主键

示例：

- `primary key (user_id, role_id)`
- `primary key (role_id, permission_id)`

## 4.4 唯一索引规范

对可空且需唯一的字段，采用 PostgreSQL 常用方式：

- 普通列允许 `null`
- 使用部分唯一索引约束非空值唯一

示例：

- `create unique index ... on users(email) where email is not null`

## 4.5 时间字段规范

- 统一使用 `timestamptz`
- 所有核心表有 `created_at`
- 所有可变核心表有 `updated_at`
- 日志表可仅保留 `created_at`

## 4.6 删除规范

- 核心业务表使用逻辑删除：`deleted_at`
- 关联表默认物理删除
- 日志表默认不可删除

## 4.7 状态字段规范

为便于迁移与兼容，状态字段采用：

- `varchar + check constraint`

不强制使用 PostgreSQL enum。

---

## 5. 对外 ID 规范

## 5.1 外部接口使用

- 接口响应返回 `public_id`
- 管理后台前端、Portal、子应用只感知 `public_id`

## 5.2 数据库内部使用

- 外键、关联、JOIN、索引使用 `id bigint`

## 5.3 映射规则

常见实体建议：

- 用户：`u_xxx`
- 部门：`d_xxx`
- 角色：`r_xxx`
- 权限：`p_xxx`
- 菜单：`m_xxx`
- Client：`c_xxx`

## 7. Better Auth 边界

以下内容归 `idp_auth`，由 Better Auth 官方 migration 管理：

- Better Auth 核心表：`user / session / account / verification`
- 插件扩展表：按当前 v1.0 计划包括 `oauthClient / oauthRefreshToken / oauthAccessToken / oauthConsent / jwks`
- 身份账号
- 认证凭证
- 授权码
- Access Token / Refresh Token 运行态
- 验证码与认证临时数据

集成约束：

- Better Auth 的 `sub` 必须能映射到 `portal_core.user_identities.subject`
- Portal 用户禁用后，IdP 不得继续允许该身份登录
- Client 配置由 `portal_core` 管理，IdP 直连只读消费
- IdP Session 固定存 Redis，不存 PostgreSQL
- Portal 创建用户时，必须同步建立 `user_identities`
- Portal 发起重置密码时，由 IdP 更新认证域
- 最终表结构必须以当前 Better Auth 配置执行官方生成命令后的结果为准

## 8. Redis 设计

## 8.1 `portal:sess:{sessionId}`

存：

- `sessionId`
- `userPublicId`
- `userInternalId`
- `createdAt`
- `lastAccessAt`
- `absoluteExpiresAt`
- `idleTimeoutSec`
- `accessToken`
- `refreshToken`
- `idToken`
- `ip`
- `userAgent`

规则：

- TTL 取 `absolute timeout`
- refresh 成功不延长绝对过期时间

## 8.2 `portal:user:sessions:{userId}`

- 存当前用户所有活跃 Portal Session ID 集合

## 8.3 `portal:auth:txn:{state}`

存：

- `state`
- `nonce`
- `codeVerifier`
- `redirect`
- `createdAt`

规则：

- TTL 建议 `5~10 分钟`
- 回调后立即删除
- `redirect` 只允许站内相对路径

## 8.4 `idp:sess:{sessionId}`

存：

- `sessionId`
- `userPublicId`
- `userInternalId`
- `subject`
- `createdAt`
- `expiresAt`

规则：

- Portal 登出必须删除对应 `IdP Session`
- 删除后不得继续支持免登

## 9. 跨架构关系

- `Portal` 读写 `portal_core.*`
- `Portal` 读写 `Redis portal:*`

## 12. 最终 DDL 落地规则

以下规则用于把本设计直接落为 PostgreSQL migration / Drizzle schema。

## 12.1 Schema 与命名

- 业务表放在 `portal_core`
- Better Auth 表放在 `idp_auth`
- 表名统一 `snake_case`
- 业务主表采用复数命名，如 `users`、`roles`、`clients`
- 关联表统一 `*_rel`
- 索引统一按 `idx_<table>_<columns>` 命名
- 唯一索引统一按 `uk_<table>_<columns>` 命名
- 外键统一按 `fk_<table>_<column>` 命名

## 12.2 主键与外键动作

- 业务主表主键统一 `bigint generated by default as identity`
- 关联表不单独生成代理主键
- `created_by`、`updated_by` 默认外键到 `users(id)`，删除动作采用 `set null`
- 主部门外键 `users.dept_id -> departments(id)`，删除动作采用 `restrict`
- 业务主数据之间的核心外键优先采用 `restrict`
- 关联表外键默认采用 `cascade`

## 12.3 逻辑删除与唯一约束

- 含 `deleted_at` 的业务表，唯一约束需考虑逻辑删除场景
- 对 `username`、`client_id`、`code` 等强唯一字段，逻辑删除后默认仍保持全局唯一，不复用旧值
- 对 `email`、`mobile` 这类可空唯一字段，采用部分唯一索引

## 12.4 时间字段维护

- `created_at` 默认 `now()`
- `updated_at` 由应用层统一维护
- 审计表和日志表默认不更新 `updated_at`

## 12.5 必建索引

- `users(username)`
- `users(dept_id)`
- `users(status)`
- `departments(parent_id)`
- `roles(code)`
- `permissions(code)`
- `menus(parent_id)`
- `clients(client_id)`
- `clients(status)`
- `audit_logs(operator_id, created_at desc)`
- `login_logs(user_id, created_at desc)`

## 12.6 JSON 与数组策略

- 业务主表不使用 JSON 存 `redirectUris / scopes / grantTypes`
- 这三类配置统一采用子表规范化建模
- 审计日志中的变更快照可使用 `jsonb`

## 12.7 migration 顺序

建议顺序：

1. 创建 `portal_core` schema
2. 创建业务主表
3. 创建关联表
4. 创建索引与唯一约束
5. 创建 `idp_auth` schema
6. 执行 Better Auth 官方生成与 migration
7. 创建 Redis key 约定对应的应用逻辑

## 12.8 Drizzle 落地要求

- Drizzle schema 只维护 `portal_core` 业务表
- Better Auth 内部表不手写到业务 Drizzle schema 中
- 若必须读取 Better Auth 表，采用单独只读 schema 映射，不把其当作业务模型扩展
- 任何 Better Auth 插件变更，都必须先更新官方生成结果，再更新文档与只读映射

## 13. 最终结论

本版数据库设计遵循常用 PostgreSQL 规范，关键冻结点如下：

- 核心表统一 `id bigint + public_id varchar`
- 关联表统一组合主键
- 外键统一引用内部 `id`
- 可空唯一字段统一用部分唯一索引
- 业务主数据、认证持久化、会话缓存分层存放

最终落位只有四类：

- `PostgreSQL.portal_core`
- `PostgreSQL.idp_auth`
- `Redis portal:*`
- `Redis idp:*`

Browser 只保留 Cookie，不承载真实认证数据。
