## 模块 H: 安全与认证 (Security & Authentication)

> 本模块统一管理所有安全相关需求，包括用户认证、会话管理、单点登录、访问控制和数据权限。

### H-AUTH: 用户认证

| ID | 需求描述 | 优先级 | 验证方法 | 风险 | 来源 | 验收标准 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **H-AUTH-001** | 未登录拦截与重定向 | 🔴关键(P0) | E2E测试 | 🔴高 | 安全审计 | 未登录用户访问任何受保护页面时，系统自动重定向到登录页面，登录成功后可跳回原始目标页面。 |
| **H-AUTH-002** | 用户登录认证 | 🔴关键(P0) | 自动化测试 / E2E测试 / 安全审计 | 🔴高 | 安全审计 | 用户输入正确的邮箱和密码后，系统验证凭证成功并建立登录会话。禁用或锁定状态的账户无法通过认证。 |
| **H-AUTH-003** | OAuth 2.1 授权码流程 | 🔴关键(P0) | 自动化测试 / E2E测试 / 安全审计 | 🔴高 | 安全审计 | 系统遵循 OAuth 2.1 Authorization Code + PKCE 标准流程签发授权码，授权码一次性使用、有效期 5 分钟。 |
| **H-AUTH-004** | 令牌安全交换 | 🔴关键(P0) | 自动化测试 / 安全审计 | 🔴高 | 安全审计 | 授权码通过服务端到服务端（Back-Channel）的方式交换为访问令牌和刷新令牌，令牌不在浏览器 URL 或前端存储中暴露。 |
| **H-AUTH-005** | 登录成功后的令牌下发 | 🔴关键(P0) | 自动化测试 / E2E测试 | 🔴高 | 安全审计 | 登录成功后，系统向浏览器下发两个 HttpOnly Cookie：访问令牌（有效期 1 小时）和刷新令牌（有效期 7 天）。 |
| **H-AUTH-006** | 浏览器 Cookie 安全 | 🔴关键(P0) | 安全审计 | 🔴高 | 安全审计 | 认证 Cookie 必须设置为 HttpOnly（不可被 JavaScript 读取）、Secure（仅 HTTPS 传输）、SameSite=Lax（防护跨站请求）。 |
| **H-AUTH-010** | 登录防 CSRF（State 参数） | 🔴关键(P0) | 自动化测试 / 安全审计 | 🔴高 | 安全审计 | 登录流程中系统生成随机 State 参数并验证回调中的 State 一致性，阻止跨站请求伪造攻击。 |
| **H-AUTH-011** | 登录防重放（PKCE） | 🔴关键(P0) | 自动化测试 / 安全审计 | 🔴高 | 安全审计 | 授权码交换令牌时，系统验证 PKCE code_verifier 与 code_challenge 的 SHA256 匹配，防止授权码拦截重放攻击。 |
| **H-AUTH-012** | 登录防重放（Nonce） | 🔴关键(P0) | 自动化测试 / 安全审计 | 🔴高 | 安全审计 | 系统在 ID Token 中嵌入随机 Nonce 值，回调时校验 Nonce 一致性，防止令牌重放攻击。 |
| **H-AUTH-013** | 登录流程超时保护 | 🔴关键(P0) | 自动化测试 | 🔴高 | 安全审计 | 登录流程的 State 参数有效期为 10 分钟，超时后需重新发起登录。 |

### H-DSCOPE: 数据范围控制

## 追溯关系

- **PRD.md**：所有需求的业务来源和优先级依据（含 CMMI Level 5 CAR 程序与过程性能基线）
- **USER_STORIES.md**：每个需求的用户故事覆盖
- **ARCHITECTURE.md**：安全需求的技术实现架构（11 层认证授权链路）
- **ARCHITECTURE_CONSTRAINTS.md**：实现层面的编码约束（领域模型校验规则等）
- **tests/traceability/coverage-report.md**：测试覆盖率报告（自动化生成，趋势受 OPM 过程性能基线监控）

## CMMI Level 5 闭环机制

- **缺陷追溯**：测试失败时通过 `@req` 注解 → 需求 ID → CAR 程序（5-Why 根因分析 → 纠正/预防措施 → 经验教训沉淀）
- **过程度量**：测试通过率（基线 100%，LCL 95%）、追溯性覆盖率（目标 85%）受统计过程控制（SPC）监控
- **持续改进**：P0/P1 缺陷触发 CAR → 预防措施更新架构约束/Red Flags → Pilot → 度量 → 推广/回滚
- **经验教训**：每次 CAR 输出写入 `docs/solutions/`，作为组织级过程资产积累
- **tests/traceability/coverage-report.md**：测试覆盖率报告（自动化生成）
