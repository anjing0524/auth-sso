# Auth-SSO 生产就绪测试与验证框架

> 日期: 2026-07-24
> 状态: 待执行
> 需求基线: `docs/spec/REQUIREMENTS_MATRIX.md`、`PRD.md`、`USER_STORIES.md`、`ACCEPTANCE_CRITERIA.md`
> 设计基线: `ARCHITECTURE.md`、`DETAILED_DESIGN.md`、`ARCHITECTURE_CONSTRAINTS.md` 与 ADR-001~009

## 1. 目标与放行原则

本框架验证的不是“测试文件存在”，而是每项已承诺需求都能由可复现的代码和强断言证据证明。

生产放行的最低条件：

1. 76 条产品需求、22 条 NFR、27 条领域约束及 R1~R14 核心架构规则均在追溯矩阵中有唯一状态。
2. 每条 P0 需求至少具有一个非 mock 的 API、浏览器或跨服务验证；安全关键需求还须有反向/攻击场景。
3. 每条需求同时具有实现证据、测试证据和执行证据。任何一项缺失均为“未验证”，不能以覆盖率补足。
4. E2E、性能、故障演练不得以 `test.skip()`、宽泛状态码断言或“无 500”断言充当通过证据。
5. 未达成的 NFR、架构约束和生产演练明确列为 Go/No-Go 阻断项或经风险接受后列为例外。

## 2. 当前基线与主要风险

截至 2026-07-24，`node tests/traceability/generate-report.mjs` 显示 76/76 产品需求有 `@req` 引用，17/27 领域约束有测试引用，27 条产品需求同时有 `@impl` 和 `@req`。这仅证明静态引用存在，不证明需求已被正确验证。

| 观察 | 风险 | 验证策略 |
| --- | --- | --- |
| 追溯脚本未解析 NFR，且将 NFR 标为未识别 ID | 100% 产品覆盖率掩盖性能与安全需求缺口 | 将 NFR 作为独立集合解析，纳入阈值、报告和 PR 门禁 |
| 49 条产品需求只有 `@req` 没有 `@impl` | 无法证明测试指向实际实现 | 为 Controller、关键领域服务、Gateway 入口和部署脚本补充准确 `@impl` |
| 部分 Playwright 用跳过、状态范围或页面未崩溃替代业务断言 | E2E 可能绿而主路径已失效 | 移除条件跳过；环境不可用即失败；断言精确 URL、Cookie、持久化数据和权限结果 |
| 当前 Portal 直连 E2E 不能证明 Gateway 行为 | HTTPS、离线验签、续签、HMAC 与路由白名单未被端到端覆盖 | 分离 Portal 浏览器 E2E 与 Gateway 全链路验收测试 |
| 无 k6 基线、故障演练、恢复演练 | NFR-PERF 和可靠性结论缺少证据 | 建立固定环境、工作负载、阈值、报告归档与发布前演练 |

## 3. 可追溯矩阵

### 3.1 矩阵结构

维护 `tests/traceability/requirements-validation.csv` 或等价的版本化 Markdown 表。每一行仅表示一个可验收原子需求，不使用“一个测试覆盖多个需求”的笼统结论。

| 字段 | 要求 |
| --- | --- |
| `id` | 产品需求、NFR、架构规则或 ADR 决策的稳定 ID |
| `source` | 规格文件和章节 |
| `priority` / `risk` | 继承需求矩阵；P0 默认阻断 |
| `acceptance` | 可观察、可断言的验收结果 |
| `implementation` | 生产文件、符号和 `@impl` ID |
| `test_id` | 稳定测试 ID，例如 `API-AUTH-003-01` |
| `level` | 静态、单元、组件、API、浏览器、Gateway、故障、性能或人工 |
| `negative_case` | 必填的拒绝、边界、重试或回滚场景 |
| `data` | 固定 seed、角色、部门层级和清理方式 |
| `evidence` | CI run、报告、截图、k6 JSON、演练记录或人工签核 |
| `owner` / `status` | 负责人及 `planned`、`implemented`、`passed`、`blocked`、`accepted-risk` |

`@req` 仅用于测试，`@impl` 仅用于生产实现。注解只可引用矩阵已有 ID；禁止以自然语言或已废弃 ID 代替需求 ID。

### 3.2 三向一致性检查

每次 PR 和发布候选版本执行以下检查：

1. 从规格解析 76 条产品需求、22 条 NFR、27 条领域约束，并单独报告 R1~R14。
2. 扫描 `apps/`、`packages/` 的 `@impl`，验证每条已实现需求可定位到具体符号。
3. 扫描测试中的 `@req` 和稳定测试 ID，验证每个测试确有针对性断言。
4. 对每个 ID 生成四态：`缺实现`、`缺测试`、`缺执行证据`、`完整`。
5. 在需求变更时强制更新对应矩阵、实现、测试和基线；删除需求或测试必须显示为显式变更，而非静默消失。

静态覆盖率只能作为发现遗漏的指标。P0 必须是 `@impl + 强测试 + 最近 CI/演练证据` 三者齐全，P1/P2 可按风险决定人工证据比例。

### 3.3 需求分组与最低测试层级

| 范围 | 最低验证组合 | 关键场景 |
| --- | --- | --- |
| A、K 门户与设计 | 组件 + 浏览器 + 视觉 + a11y | 菜单按权限隐藏、面包屑、看板范围、移动端、空状态 |
| B、C、D、E、F RBAC 管理 | 领域 + API 真实持久化 + 浏览器 | 唯一性、事务回滚、逻辑删除、循环部门、角色部门约束、权限即时生效 |
| G 客户端管理 | 领域 + API + OAuth 集成 | Secret 仅展示一次、轮换后旧 Secret 失效、客户端注销撤销令牌 |
| H-AUTH、H-SESS、H-SSO、H-ACL | 领域 + API + 浏览器 + Gateway | OAuth 2.1/PKCE、Cookie、轮换、撤销、SSO、403 与数据范围 |
| J 审计日志 | API + DB/部署校验 + 浏览器 | 关键事件写入、不可 UPDATE/DELETE、筛选、导出、180 天保留 |
| NFR-SEC | 静态 + 攻击 API + Gateway/Rust | ES256 锁定、State/Nonce、重放、重定向、速率与暴力破解 |
| NFR-PERF | k6 + Rust benchmark | `/api/me`、分页、登录旅程、500 并发、Gateway 验签延迟 |
| R1~R14、ADR | 静态规则 + 定向集成 | 分层、事务、枚举来源、缓存、三层鉴权、Gateway 路由与刷新 |

## 4. 分层测试设计

### T0: 文档、静态与部署验证

在无需运行服务的情况下验证规格到实现的结构性承诺：

- 追溯脚本必须解析产品需求、NFR、领域约束和 ADR 验收项，拒绝未知 ID。
- 以 AST 或受控 `rg` 规则检查 R1~R14：Controller 行数、`mapDomainError()`、多写事务、contracts 枚举来源、禁止 domain import `next/*`/Drizzle、`data.ts` 缓存约束。
- 解析 Drizzle schema 与 migration，核对 `DATABASE.md` 的表、索引、约束与分区策略。
- 检查审计表无应用层 UPDATE/DELETE 路径，并在临时数据库以应用角色验证 `REVOKE UPDATE, DELETE`。
- 检查 Portal、Gateway、Demo 的环境变量、TLS、Cookie、安全头和共享 JWT/HMAC 契约。

### T1: 领域单元测试

覆盖无基础设施依赖的规则，并优先采用表驱动边界测试：

- 用户状态、密码策略和最近 5 次密码历史。
- 部门移动的自身、子节点、深层子节点循环，及祖先路径级联。
- 角色部门归属、系统角色保护、角色删除关联关系。
- OAuth 授权码 TTL、一次性领取、PKCE S256、State、Nonce、scope allow-list、redirect URI 精确匹配。
- Refresh Token 轮换与旧令牌立即失效。
- contracts 的 TTL、权限码、枚举与 Portal/Gateway claims 兼容性。

每个规则至少包含成功、边界、拒绝三个类别。对树、集合、权限并集和 token 状态机加入性质测试或生成式输入，验证不变量而非只验证少量示例。

### T2: Portal API 与 Server Action 集成测试

使用迁移后的 PostgreSQL 和 Redis，不 mock 被测业务边界；可 mock 外部网络和时间，但不得 mock DB 写入、事务或鉴权决策。

- CRUD 测试必须在响应后查询真实数据库，验证持久化值、关联行、审计行与回滚。
- 每个受保护端点须有 `401`、`403`、本部门允许、跨部门拒绝和管理员允许的测试。
- 写操作须验证失败时没有半完成数据，尤其是角色分配、权限变更、删除、令牌撤销和审计写入。
- API 契约测试验证状态码、`ApiSuccess<T>/ApiError` 结构、分页、校验错误与秘密字段不泄露。
- Redis 不可用时分别验证 Portal 的规定降级路径和安全关键操作的 fail-closed 行为。

测试隔离使用事务回滚或唯一前缀数据，并在并发执行前证明隔离可靠；不得依赖跨测试的顺序或残留 seed。

### T3: 组件、可用性与无障碍

组件测试验证用户可见行为，不验证实现细节：

- 表单必填、格式、服务端错误映射、pending 禁用与双击提交防护。
- `PermissionGuard`、侧栏、数据表、空状态、命令面板、Dialog 的键盘与焦点行为。
- 角色变更、删除、锁定等高风险操作必须有确认、结果反馈和失败恢复。
- 使用 `@axe-core/playwright` 或等价工具检查关键页面；人工复核键盘流、焦点可见、语义标签、`aria-live` 与文字对比度。

### T4: Portal 浏览器 E2E

浏览器测试以真实的 Portal、PostgreSQL、Redis 和固定 seed 运行，覆盖用户旅程：

1. 未登录访问受保护页，登录后精确返回原始安全路径。
2. 正确和错误凭据、锁定账户、错误提示与状态恢复。
3. OAuth Authorization Code + PKCE：浏览器发起、回调 state/nonce、Cookie 下发、Dashboard 渲染。
4. 登出、强制下线、密码重置、角色撤销后的下一请求失效。
5. 管理员创建、禁用、删除用户；角色和权限变更后的菜单、页面和 API 同步变化。
6. 数据范围以至少三层部门树和多角色并集验证。

禁止在认证、数据创建、重定向或授权码签发失败时跳过。测试前置条件不满足时应使 job 失败，并由全局 setup 给出诊断。

### T5: Gateway 与跨服务验收

独立于 Portal E2E，启动 Gateway HTTPS、Portal、PostgreSQL、Redis 和 Demo 应用，所有请求经 Gateway：

- 未认证 HTML 导航必须进入 PKCE；API 请求返回预期的未认证响应。
- ES256 JWT 的 `alg`、`kid`、issuer、过期、签名篡改和 jti 黑名单由 Gateway 离线验证。
- 临近过期 AT 使用 RT 续签，Gateway 透传新 Cookie，RT 不泄漏到上游。
- Gateway 到 Portal 的刷新 HMAC 通过；伪造和缺失签名被拒绝。
- 路由白名单、静态扩展名、上游 TLS、客户端 IP 以及认证限流符合 ADR-009 与 Gateway 配置。
- Portal 已登录到 Demo 免登，登出后双方会话均不可再用。

### T6: Rust Gateway 单元、集成与基准

`cargo test --all-targets` 覆盖 `AuthDecision`、验签、JWKS 轮换、Cookie 重写、路径分类、限流、Redis 错误和 refresh 协调。所有自定义异步 Trait 增加编译期或代码审查检查，确保 `impl Future + Send`，不引入 `#[async_trait]`。

保留 Criterion 热路径基准，并增加可比的基线文件。任何 PR 中 Gateway 验签、路径分类、Cookie 或 JWKS 基准退化超过预设预算都需人工批准。

### T7: 安全、可靠性与恢复

安全测试必须构造有效前提后再验证拒绝结果，例如先签发真实授权码，再重复兑换或提供错误 verifier；不能仅提交一个随机无效 code。

| 演练 | 通过标准 |
| --- | --- |
| Redis 不可用 | 明确区分 Gateway jti 检查的既定 fail-open 与 Portal 权限读取的既定降级；无未捕获异常或静默放权 |
| PostgreSQL 中断/恢复 | 请求在超时预算内失败且不泄露内部信息；恢复后新请求正常 |
| Gateway/Portal 重启 | 健康检查恢复、无持续 5xx、会话及续签行为符合设计 |
| JWKS 轮换 | 新旧密钥过渡期验证成功，过期 kid 按策略刷新或拒绝 |
| 事务和审计写入失败 | 业务操作与审计要求保持原子性；不得产生未审计的安全关键变更 |
| 备份恢复 | 以演练备份恢复到隔离库，核对用户、RBAC、令牌撤销与审计记录 |

### T8: 性能与长期稳定性

使用版本化 k6 脚本、固定数据量和隔离环境建立以下基线：

| NFR | 工作负载 | 通过阈值 |
| --- | --- | --- |
| NFR-PERF-01 | `/api/me` 稳态读 | P95 < 200ms |
| NFR-PERF-02 | 浏览器登录至 Dashboard | P95 < 1.5s |
| NFR-PERF-03 | Gateway ES256 离线验签 | < 5ms，不含网络 |
| NFR-PERF-04 | 1,000 条列表分页 | P95 < 500ms |
| NFR-PERF-05 | 500 并发用户 | 错误率 < 1% |

发布候选版本至少执行一次 60 分钟稳定性运行；重大 Gateway、缓存、认证或数据库变更执行 24 小时 soak。报告必须包含版本、环境、数据量、并发、p50/p95/p99、错误率、资源使用与结论。

## 5. UI/UX 验证方法

视觉快照用于发现漂移，不是设计正确性的唯一证明。对 `/login`、Dashboard、用户/角色/客户端/审计页面、403/404、移动端和深色模式维护经人工批准的截图基线。

每次视觉基线更新需要设计或产品签核，说明变更原因。浏览器用例还须执行：

- 关键任务走查：登录、创建用户、分配角色、撤销权限、导出审计日志。
- 错误预防：危险操作确认、不可用操作隐藏或禁用、输入即时反馈、提交中不可重复发起。
- 恢复能力：网络断开、服务错误、会话过期时给出可执行提示，不出现空白页。
- 无障碍：键盘完成核心任务，焦点顺序和错误宣告正确，关键控件具有可访问名称。

## 6. 执行顺序

1. 修正追溯工具：解析 NFR、拒绝未知 ID、输出实现/测试/执行证据三向状态；为所有 P0 与缺失产品需求补 `@impl`。
2. 整治弱测试：移除跳过和宽泛断言，改用真实前置数据、精确安全断言与数据库读回验证。
3. 补齐 T0/T1/T2：先覆盖事务、审计不可变、授权码、刷新、RBAC 和数据范围的边界。
4. 将 Portal E2E 与 Gateway 全链路验收分开运行，CI 中分别提供环境与日志工件。
5. 增加 a11y、视觉评审、故障演练、k6 与备份恢复证据。
6. 按矩阵执行发布候选验收，召开 Go/No-Go 审查；只允许将已记录、已批准、有到期日的残余风险带入发布。

## 7. CI 与发布门禁

| 阶段 | 必须通过 |
| --- | --- |
| 每个 PR | lint、typecheck、Vitest、contracts、静态架构规则、追溯完整性、Cargo fmt/clippy/test/audit |
| 合并主干 | 上述全部 + Portal E2E + Gateway 全链路验收 + 视觉/a11y 冒烟 |
| 发布候选 | 上述全部 + 追溯矩阵全量审计 + 故障演练 + k6 基线 + JWKS 轮换 + 备份恢复 |

门禁阈值：

- P0：100% `完整`，无跳过、无风险接受替代自动化验证。
- 产品需求：100% 具备实现与测试；P1/P2 的人工验证须有签核证据。
- NFR：100% 具备测试或演练；性能阈值按 PRD 原值执行。
- 架构：R1~R14 全量静态检查，27 条领域约束 100% 有定向验证。
- 测试结果：重试后的通过不掩盖首轮失败；任何 flaky 测试先隔离为失败并创建缺陷，再恢复门禁。

## 8. 交付物与责任

每次发布生成以下版本化或 CI 归档工件：

1. 需求验证矩阵及追溯报告。
2. Portal API、浏览器、Gateway、Rust 的测试报告和失败日志。
3. 视觉 diff、a11y 报告与人工 UX 走查记录。
4. k6 原始结果与性能对比。
5. 故障、密钥轮换、备份恢复演练记录。
6. Go/No-Go 清单：阻断项、已接受风险、责任人、到期日和回滚条件。

QA 负责矩阵完整性和证据质量；开发负责实现映射与自动化修复；安全负责人签核攻击与密钥演练；运维负责人签核容量、恢复与监控证据；产品/设计负责人签核关键旅程和视觉基线。
