# 事务化角色绑定与安全审计

**严重等级**：P0 阻断 / P1 一般  
**发现途径**：代码审计

## 问题现象

用户角色绑定在 transaction 回调内校验角色，但校验函数通过全局 `db` 读取角色；角色部门或状态并发改变时，校验与写入不再是同一事务事实。安全审计同时以 fire-and-forget 方式在业务提交后调度，成功响应不保证有对应记录。

## 根因分析

### 5-Why

1. 为什么可能写入不合法角色？校验读取没有使用 `tx`。
2. 为什么会使用全局连接？校验函数的依赖没有显式声明。
3. 为什么安全审计会遗漏？守卫把审计视为通用后置副作用。
4. 为什么后置副作用可丢失？没有区分合规审计和访问观测。
5. 为什么没有测试发现？角色绑定路由缺少事务和异常输入集成测试。

### 根因陈述

事务边界和审计交付语义未被编码为明确接口，导致读取连接与审计持久化都可以被调用方绕过。

## 纠正措施

- `validateRoleAssignment(tx, ...)` 只通过当前 transaction 查询角色，并验证角色集合完整性、部门和 ACTIVE 状态。
- 角色绑定输入使用 Zod 校验 UUID、数量和重复值；移除不存在的 `user_roles.id` 写入。
- `appendSecurityAudit(tx, ...)` 在角色绑定 transaction 内执行；审计失败使事务回滚。
- `mapDomainError()` 不再依赖 logger；REST 守卫统一输出 `{ error, message }`。
- 权限 REST 列表改为 SQL `count + limit + offset` 分页。

## 预防措施

- 任何“读取事实后写入授权关系”的代码，读取必须显式接收并使用 `tx`。
- 合规审计仅允许 transaction 内 append；访问日志才允许异步 best-effort。
- Route body 必须以 Zod schema 完整解析，数组输入必须验证类型、上限和唯一性。
- 为角色绑定维持非法输入、集合完整性和审计失败回滚测试。
