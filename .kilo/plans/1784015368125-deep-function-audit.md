# 全维度深度函数级代码审计计划

> **制定日期**：2026-07-14
> **项目**：auth-sso（Portal + Gateway + Contracts + Config）
> **审计侧重**：函数级深度审计 — 消除歧义、消除重复、消除雍容、真实测试、真实业务逻辑统一
> **背景**：前 3 轮审计（7/10、7/13、7/14）已覆盖架构/Top10 安全/错误码统一/CI/CD 等高阶问题，本轮聚焦函数级细节

## 一、审计策略

### 1.1 规模适配

| 项目 | 代码量 | 
|------|--------|
| Portal（TS/TSX 源码） | ~19,500 行 |
| Portal（测试） | ~6,700 行 |
| Gateway（Rust） | ~4,600 行 |
| Contracts + Config | ~1,000 行 |
| Demo App | ~500 行 |
| **合计（排除 node_modules/target/.next）** | **~43,000 行** |

→ 中型规模 → **2 路 Agent 并行**

### 1.2 Agent 分工（与前 3 轮互补，避免重复审计）

#### Agent A：函数质量与消除歧义/重复/雍容

覆盖角色：角色7（Clean Code）+ 角色3（架构）+ 角色14（业务治理）

审计重点：
1. **函数级重复代码检测**：跨模块 CRUD Action 是否存在重复模式（`users/actions.ts` vs `departments/actions.ts` vs `roles/actions.ts` vs `clients/actions.ts` vs `permissions/actions.ts`）
2. **函数职责拆分**：大型函数（>40 行）是否违反单一职责
3. **命名歧义**：变量/函数命名是否与语义一致（"session" 多义、`result` vs `response`、`data` 混用等）
4. **冗余抽象**：是否存在过度包装（单次调用的工厂函数、不必要的间接层）
5. **错误处理歧义**：不同模块对同类错误的处理是否一致
6. **魔法值散落**：硬编码常量是否已收敛（password length、brute force threshold 等）

文件发现策略：
- 所有 `actions.ts` 文件（5 个 — users, departments, roles, clients, permissions）
- 所有 `route.ts` 文件（API 端点，约 15 个）
- `lib/` 下所有 TypeScript 文件（约 20 个核心工具/业务文件）
- `domain/` 下 6 个模块的业务逻辑文件
- 跨模块搜索重复代码片段（通过结构相似性分析）

#### Agent B：测试真实性与业务逻辑一致性

覆盖角色：角色12（测试深度）+ 角色6（全链路）+ 角色1（需求工程）+ 角色4（数据建模）

审计重点：
1. **测试真实性验证**：逐个检查测试断言是否验证了业务行为（而非仅 `status === 200` 或 `r.success === true`）
2. **测试用例完备性**：核心函数是否有边界/异常/空值测试
3. **业务逻辑一致性**：同一个业务规则在不同模块中的实现是否一致（如 dept 数据范围校验、权限继承规则）
4. **Domain 层测试覆盖率**：哪些 domain 纯函数缺少测试或测试薄弱
5. **Mock 污染检测**：API 测试中的 Mock 是否使得测试完全不验证真实行为
6. **测试文件与源码对应关系**：是否存在有源码无测试的核心模块

文件发现策略：
- 17 个 API 测试文件（`__tests__/api/*`）— 逐文件审查断言有效性
- 7 个 Domain 测试文件（`__tests__/domain/*`）— 检查覆盖缺口
- `tests/integration/` — 检查实际覆盖的端点和场景
- 对照 domain 层源码文件，检查哪些纯函数缺少测试

**测试真实性评判标准**：
- ❌ 无效测试：仅断言 `response.status === 200` + `body.data` 存在，不验证数据内容
- ⚠️ 弱测试：断言了数据字段存在但不验证业务正确性
- ✅ 有效测试：断言了具体业务行为结果（如"用户状态从 ACTIVE 变为 LOCKED"）

### 1.3 排除范围（前 3 轮已充分覆盖）

以下领域本轮不重复审计（除非发现回归）：

- API 响应格式策略（已在 7/14 二次审计中定论：REST 用 HTTP 状态码，Server Action 用 ApiResponse）

## 二、执行步骤

### 2.1 第一步：全局分析 Explore Agent

输出「项目结构适配报告」，包含：
- 四层服务角色映射（前端入口 / 后端 API + 业务层 / 共享契约 / 基础设施）
- 关键路径映射（actions.ts 清单、route.ts 清单、domain 文件清单、test 文件清单）
- 核心业务链路简述

### 2.2 第二步：2 路 Agent 并行审计

**Agent A（函数质量）** 和 **Agent B（测试真实性）** 同时启动。

每个 Agent 必须：
1. 先 Read 对应的 reference 文件获取详细检查清单
2. 利用第一步的路径映射定位实际文件
3. 每个角色至少输出 5 条具体发现
4. 每条发现必须包含 `file:line` + 风险等级 + 影响 + 判断依据

### 2.3 第三步：复核 Agent

交叉验证 2 路 Agent 的全部输出：
- 标记误报和证据不足的发现
- 去重合并（两个 Agent 可能从不同角度发现同一问题）
- TOP10 最严重问题逐条二次 Read 代码确认
- 盲区补充

### 2.4 第四步：输出最终报告

按 6 部分固定结构输出（见下）。

### 2.5 第五步：文档落地

写入 `docs/audit/2026-07-14-deep-function-audit.md`。

## 三、风险控制

- 本轮审计**不修改代码**，仅产出诊断报告和修复建议
- 所有建议必须保证业务逻辑 100% 等价
- 证据驱动：每个问题必须有精确的 `file:line` 引用
- 零新增依赖
- 复核 Agent 必须对 TOP10 严重问题二次 Read 确认

## 四、成功标准

1. 发现 ≥10 个前 3 轮未覆盖的函数级问题（歧义/重复/雍容/测试不真实）
2. 每个 API 测试文件都有「测试真实性评级」（有效/弱/无效）
3. 识别 ≥3 处跨模块代码重复，给出合并方案
4. 识别 ≥3 个测试覆盖盲区（有源码无测试的核心函数）
5. 最终报告文件落盘
